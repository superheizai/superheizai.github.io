git回滚操作

在日常工作中，git的回滚操作，主要接触到了三种，分别是本地修改，本地提交(commit)和远程提交(push)。就这三种不同的场景，分别记录下不同的回滚方式。

本地修改

本地修改的内容只是体现在当前文件中，如果未保存而关闭了编辑器的话，那么当前的修改会丢失，找不回来。
对本地修改的回滚操作是通过checkout命令来完成的

  • 某个特定文件：git checkout filename
  • 全部文件：git checkout .

本地提交(commit)

本地已提交，可以通过commit log(git log或者git reflog)查看local repo里面代码提交记录，但是在远程repo不存在相关提交。
对于本地提交的回滚是通过reset操作来完成的

reset三种模式

参考高飞鸟博客
参考廖雪峰博客

  • mixed（默认），它回退到某个版本，只保留源码，回退commit和stage信息
  • soft，回退到某个版本，只回退了commit的信息，不会恢复到stage file一级。如果还要提交，直接commit即可
  • hard，彻底回退到某个版本，本地的源码也会变为上一个版本的内容

使用例子

//回退所有内容到上一个版本
git reset HEAD^
//回退a.py这个文件的版本到上一个版本
git reset HEAD^ a.py
//向前回退到第3个版本
git reset –soft HEAD~3
//将本地的状态回退到和远程的一样
git reset –hard origin/master
//回退到某个版本
git reset 057d

远程提交(push)

参考韩小虎的技术博客
参考一介布衣博客

对于远程提交来说，代码已经同步到远程仓库，这个时候在本地用reset是可以把本地的分支回滚到某个时间点。但是，在做代码合并的时候仍然会从远程仓库里面拿到之前已经提交过的修改。
revert和reset针对本地commit的区别，revert会产生一个针对此commit的逆操作，而reset是撤销这个操作。revert可以解决这个问题。而且revert只是针对单词commit的逆操作，而reset是直接到回滚到第n个commit。revert和reset还有其它的一些区别，这里不再详述。

回滚某次提交

git revert commitid

这种情况最常见，是说针对某个文件做了错误的修改，你想恢复之前的状态，那么可以直接revert这个commit，然后再做相应的修改，然后再次提交。这里的revert会把所有的修改revert掉，所以相关的操作最好一起做commit。

回滚某次merge

git revert -m 1 commitId

回滚某次merge的最主要的点是，你要提供出你要回滚到出现本次merge的commit，你要回滚到的分支commit出现的顺序，比如语法中使用的1.

当你使用 git revert 撤销一个 merge commit 时，如果除了 commit 号而不加任何其他参数，git 将会提示错误：

$ git revert 83281a8e9aa1ede58d51a6dd78d5414dd9bc8548 //本人实际git信息，这里对应git演进图中的 g
error: Commit g is a merge but no -m option was given.
fatal: revert failed

在你合并两个分支并试图撤销时，Git 并不知道你到底需要保留哪一个分支上所做的修改。从 Git 的角度来看，master 分支和 dev 在地位上是完全平等的，只是在 workflow 中，master 被人为约定成了「主分支」。

于是 Git 需要你通过 m 或 mainline 参数来指定「主线」。merge commit 的 parents 一定是在两个不同的线索上，因此可以通过 parent 来表示「主线」。m 参数的值可以是 1 或者 2，对应着 parent 在 merge commit 信息中的顺序。

以上面那张图为例，我们查看 commit g 的内容：

$ git show 83281a8e9aa1ede58d51a6dd78d5414dd9bc8548 //本人实际git信息，这里对应git演进图中的 g

commit 83281a8e9aa1ede58d51a6dd78d5414dd9bc8548

Merge: 312a518 fa87415 //312a518和fa87415 可以在git log中找到对应的提交信息（只是commit一长串字符的头部分）
……

那么，$ git revert -m 1 g 将会保留 master 分支上的修改，撤销 dev 分支上的修改。//(1就是1，表示312a518对应的父来源，2表示fa87415对应的父来源)撤销成功之后，Git 将会生成一个新的 Commit，提交历史就成了这样：

a -> b -> c -> f -- g -> h -> G (master)
              \      /
                   d -> e  (dev)

其中 G 是撤销 g 生成的 commit。通过 $ git show G 之后，我们会发现 G 是一个常规提交，内容就是撤销 merge 时被丢弃的那条线索的所有 commit 的「反操作」的合集。
